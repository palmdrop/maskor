About structure before writing: I agree. It is a good idea to let users see the structure before maskor tries to help them create it. The core mechanism should be: users write and arrange fragments, they see how aspects rise and fall. They can use that shape to generate arcs, or draw their own and see the "gap" between their fragments aspects and the drawn arcs. I still think we should keep the arc concept as is, but one important addition would be a way to view the "ACTUAL" arc that is formed by the fragments. Maskor should let the user inspect the gap, help them shape the arcs they want, and assist in placing new fragments in this configuration. However, it should not force users to define arcs before doing anything else. A user can use maskor for a complete project without ever creating an arc, but instead just inspecting the actual shape of the story. One idea here is to have "implicit" arcs, where maskor uses the actual shape of fragments to place new ones (if explicit arcs are not defined). I REALLY LIKE THIS IDEA. Mention this in `aspect-arc-model.md`, but keep the existing arc concept as is.

About AI: I'm highly critical to AI being used as a tool in actual writing. I do not want it. Your concern about control points is not valid: the user will never define actual coordinates, they will draw the arc in a canvas-like view or derive it from their sequence.

About orphan-warnings: Maskor should help with renames. If a user renames a note or reference, this should update all the fragments that use it. This should happen automatically, regardless if the rename happens through maskor or in the vault directly. However, if a note is renamed when maskor is not running, then there is no way to know what the new file is. Then, a warning is appropriate.

About the prompting mechanism: Thanks for reminding me. I agree, it is one of the core concepts. Lets create a spec for it and reference other relevant specs.

About undo-redo: Yes, lets keep it simple. I like your suggestion. The core principle should probably be this: just keep an action log for observability. During a session, undo-redo should work for fragment edits, sequence re-ordering, etc. Common, sensible actions, where the user expects undo-redo to be available. Across sessions, the undo-redo stack can be dropped for now.

About the user: I agree that most users will use the bottom-up approach. The sequencing-helper is an optional feature that many users will not benefit from. However, maskor is opinionated, and it is personal. It should of course not force semi-automatic, semi-random sequencing and suggestions onto the user, but I personally like this aspect, and I'm intrigued by soft patterns and randomness, and think it is a great tool that can create interesting creative effects. It is important to me, and should remain a concept in maskor.

With these notes in mind, update the specs to reflect this.

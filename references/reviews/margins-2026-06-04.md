**Date**: 2026-06-04
**Scope**: `packages/frontend/src/components/margins/`, `prose-editor.tsx`, `anchor-{cm,tiptap}.ts`, `block-spacer-{cm,tiptap}.ts`, `hooks/useMarginEditor.ts`, `lib/margins/`, `packages/shared/src/utils/comment-marker.ts`, `packages/storage/src/{service,watcher,vault,indexer}` margin code, `packages/api/.../margins`

**Spec**: `specifications/margins.md`, `specifications/fragment-editor.md`
**ADRs**: 0007, 0008, 0009

---

> **Resolution (2026-06-04): all ten findings fixed.** #1 stored excerpt is now backend-authoritative (`persistMargin` re-derives anchored excerpts on every write). #2 the per-comment CRUD + `listOrphanedComments` API was deleted (routes/commands/schemas/storage/indexer query + `comment:*` action types); per-comment DB rows kept as the internal index, and on a follow-up the now-readerless `orphaned` column + `recomputeMarginOrphans` were also dropped (migration `20260604_drop_comment_orphaned`) — orphan state is purely live-derived, `margin:synced` fires only on a real excerpt change. #3 the marker-set derivation unified via shared `markerIdSet`. #4 `margin:save` removed. #5 ↓/↑ slot navigation implemented (caret-boundary detection per editor). #6 fuzzy rebind gated on `fragmentDirty`. #7 orphan group suppressed until first block measurement. #8 orphans edit through `SlotEditor` (the `CommentCard`/textarea was deleted). #9 `enumerateBlocks` moved to a test-only helper. #10 handle methods renamed off the `*CommentMarker` names. Specs/ADR/suggestions reconciled; `bun run verify` green.

## Overall

The feature is large and largely coherent. The ADR-0009 cutover (buffer-clean anchoring, editor-driven geometry, mutual flow alignment) is implemented cleanly: anchors are real PM positions / CM offsets mapped through transactions, markers strip-on-load and re-emit-on-save, and the deletion-drop logic (margins-4 #7) is correct in both editors. The backend storage/index/watcher path is solid and well-tested.

The most important finding is a **coupled-save ordering bug** that quietly defeats the spec's "stored excerpt refreshes on save" invariant whenever the fragment and Margin are saved together — which is the common case. Beyond that, the main issues are accumulated dead/divergent surface: a whole per-comment CRUD API that no client uses, a specced keymap branch that was never wired, and several stale handle names/duplications already half-acknowledged in `suggestions.md`.

## Bugs

### 1. Coupled save clobbers the backend-refreshed excerpt with the frontend's stale one

`packages/frontend/src/components/fragments/fragment-editor.tsx:187` (`onContentSave`) saves in order: fragment first (when `isProseDirty`), then Margin (when `marginEditor.isDirty`).

```
fragment save  → backend refreshMarginExcerptsOnFragmentSave()
                 rewrites each anchored comment.excerpt = current block opening, persists margin file + DB
margin  save   → writeMargin PUT { comments }  (frontend Comment objects)
                 persistMargin() rewrites the file with the FRONTEND excerpts → overwrites the refresh
```

The frontend only ever sets `comment.excerpt` at creation time (`addCommentStub` in `margin-column.tsx:329`); `updateCommentBody` never touches it, and nothing re-derives it from the live block. So a comment whose block opening was edited after the comment was created carries a stale stored excerpt, and the coupled save writes that stale value back over the backend's just-refreshed one.

Consequences:

- The Obsidian-visible `> excerpt` drifts from the block it annotates (spec § "stored excerpt refreshes from the block's current opening on fragment save" — `specifications/margins.md:79`).
- When that comment is later orphaned the **frozen** excerpt is the stale one, so `planOrphanRebinds` (`lib/margins/column.ts:64`) matches `deriveExcerpt(block.text)` against the wrong text — the delete→paste re-attach path (margins-4 #7, the headline fix) silently fails for exactly the blocks that were edited after commenting.

This only bites when fragment + Margin are dirty in the _same_ save (the normal flow). A fragment-only save refreshes correctly because the margin PUT doesn't run.

Fix: don't have the frontend own the excerpt on the coupled save — either drop `excerpt` from the `writeMargin` payload and let the backend derive/refresh it, or run the margin write _before_ the fragment save so the backend refresh wins, or have the column keep `comment.excerpt` live (re-derive from the bound block on each render before serialize).

## Design

### 2. The per-comment CRUD API and `listOrphanedComments` are dead from the only client

`packages/api/src/routes/margins.ts` exposes `createComment` / `updateComment` / `deleteComment` / `listOrphanedComments`, each backed by a command (`packages/api/src/commands/margins/*`) and storage method. The frontend uses **none** of them — `useMarginEditor` reads `getMargin` and writes the whole document via `writeMargin` (PUT replace), and orphans are derived live in `margin-column.tsx`/`column.ts`, never fetched. (Confirmed: no non-generated reference to `useCreateComment`/`useUpdateComment`/`useDeleteComment`/`useListOrphanedComments`.)

These were Phase-2 deliverables and are backend-tested, but they are now maintained surface with no consumer. The 1:1 marker↔comment model plus the replace-on-save client makes granular CRUD redundant. Either delete them (and the schemas/commands) or document them explicitly as an external/Obsidian-facing API so the divergence is intentional, not accidental.

### 3. Duplicated, divergent `loadFragmentMarkerIds`

`packages/storage/src/service/storage-service.ts:200` and `packages/storage/src/watcher/sync/margin.ts:17` both implement "the marker ids in a fragment's body, by uuid, null when unknown" — but differently: the service reads through `vault.fragments.read` (the mapper), the watcher reads the file bytes directly with `Bun.file`. Same contract, two implementations, two failure modes to keep in sync. `.claude/CLAUDE.md` calls this out ("Create re-usable functions. If you notice overlap, break out into a new function."). Extract one shared helper.

### 4. `margin:save` palette command is not coupled and is now inconsistent

`packages/frontend/src/lib/commands/scopes/margin.ts:15` still saves only the Margin (`handleMarginSave`), while the dedicated Margin Save button is gone and `editor:save` is the coupled path (margins-4 #13). A user running "Save margin" from the palette persists the Margin but not the dirty fragment, leaving a marker un-re-emitted until the next fragment save. Harmless (orphan-tolerant), but a footgun now that it's the only non-coupled save. Already flagged in `suggestions.md` — make it delegate to `editor:save` or drop it from the palette.

## Minor

### 5. ↓/↑ slot navigation is specced as shipped but never wired

`specifications/margins.md:117` and the margins-2/4 Shipped notes claim "Tab/Shift-Tab (and ↓/↑ at comment boundaries) move between slots." `SlotEditor.handleKeyDown` (`slot-editor.tsx:70`) only intercepts `Tab` and `Escape`; `onNext`/`onPrevious` are wired to Tab alone. Arrow-at-boundary navigation does not exist. `suggestions.md` (2026-06-02) already notes the gap, but the spec still asserts it as shipped — fix one or the other.

### 6. Orphan re-bind can dirty the fragment merely by opening it

The rebind effect (`margin-column.tsx:288`) calls `insertMarkerInBlock` whenever a stored orphan's excerpt uniquely matches an un-anchored block, which fires `onChange` → marks the fragment dirty. On a fresh load of a fragment that has such an orphan, just opening it lights the Save button and stages a marker the user never asked for. Intended for the delete→paste case, surprising as a load-time side effect. Consider gating rebind to user-initiated edits, or accepting it but documenting the auto-dirty.

### 7. Transient "everything is orphaned" flash on mount

`editorBlocks` comes from `getBlocks()`, which returns `[]` until the editor is laid out (`prose-editor.tsx:512` / shell ref not yet populated). For the first frame `buildColumn` sees no blocks, so every anchored comment lands in `orphans` and the orphan group renders, then vanishes once `geometryTick` re-pulls. Cosmetic, but visible. Guard the orphan group on a "geometry measured" flag, or render nothing until the first real `getBlocks()`.

### 8. Orphan editing uses a plain `<textarea>`, breaking the seamless-serif model

Orphans render via `CommentCard` (`comment-card.tsx`), whose expanded body is a `<textarea>` with `border-input` / `text-sm` — not the mode-coupled `SlotEditor` the rest of the column uses (margins-4 #1/#8 made everything serif flowing text). Also its excerpt button calls `onReveal`, which the column never passes for orphans (`margin-column.tsx:498`), so it's a dead click with a misleading `title="Anchor lost"`. Either route orphans through `SlotEditor` or trim `CommentCard` to a read-only orphan card.

### 9. Production-dead `enumerateBlocks`

`lib/margins/column.ts:16` is now only referenced by tests — the render path binds off `getBlocks()` (ADR 0009). It's documented as a "test-harness fallback," which is fine, but it's exported as if it were API. Demote/comment it as test-only or fold the test to use `getBlocks` stubs.

### 10. Stale handle names (carried, acknowledged)

`prose-editor.tsx` still exposes `appendCommentMarker` / `insertCommentMarkerInBlock` / `stripCommentMarker` / `revealCommentMarker` / `focusMarkerBlock`; none touch buffer markers any more (they manipulate the anchor store). Already in `suggestions.md`; noting so the rename isn't lost. The whole shell→column prop chain mirrors the misleading names.

## Non-issues

- **`serializeTiptapWithMarkers` swallows insert errors and drops a marker** (`anchor-tiptap.ts:75`) — intentional: a drifted position that can't host an inline node degrades to an orphan rather than throwing on save. Acceptable per ADR 0009's "fuzzy/recovery" stance.
- **Scroll-sync `syncing` guard looks ineffective** (`margin-column.tsx:179`) — scroll events fire async so the synchronous flag never spans the echo, but assigning an unchanged `scrollTop` emits no further event, so there's no ping-pong. Fine.
- **Two scrollers mirrored by raw `scrollTop` despite different total heights** — by design (ADR 0009 rejected a single physical scroller); congruence comes from the unified block source + document-side spacers, and the margin's extra bottom content (notes) sits past the aligned region.
- **`addCommentStub` is idempotent / reseeds excerpt on re-run** (`useMarginEditor.ts:135`) — matches the 1:1 marker↔comment rule; re-gesturing a block won't duplicate.
- **`dedupeComments` keeps first position, last content** (`mappers/margin.ts:67`) — deliberate guard against an external edit duplicating a marker block colliding on the `(fragmentUuid, markerId)` PK.
- **Document-side spacers are decorations/meta-only transactions** (`block-spacer-{cm,tiptap}.ts`) — confirmed they never set `docChanged`, so they don't dirty the buffer or serialize.
- **`notes:` removal in `update-fragment.ts`** — the deleted `fragment-update-changedfields.test.ts` and the removed note-attach/detach log entries are the intended Phase 8 drop, not a stray regression.

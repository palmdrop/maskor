# Sync Contract

Defines the boundary between **markdown files** (user-owned) and the **database** (Maskor-owned).

The database is a cache and index derived from the vault files. It can always be fully rebuilt from the vault. Markdown is the source of truth for everything it owns.

---

## Vault Structure

```
<vault>/
  fragments/            # Fragment files
  fragments/discarded/  # Discarded fragments
  aspects/              # Aspect definitions
  notes/                # Note files
  references/           # Reference files
  pieces/               # Consume directory — raw pieces dropped here are converted to fragments
```

---

## Fragments

**File:** `fragments/<title>.md`

Each fragment file has a YAML frontmatter block, optional inline aspect fields, and a markdown body.

Discarded fragments are moved to the `fragments/discarded` folder. This overrides the `pool` property, however the `pool` property should automatically be updated. Moving a discarded fragment out of the `discarded` folder should automatically update the `pool` property.

### Frontmatter (markdown-owned)

Maps to fields on the `Fragment` type.

| Frontmatter key | Type field             | Notes                                                                                       |
| --------------- | ---------------------- | ------------------------------------------------------------------------------------------- |
| `uuid`          | `uuid: FragmentUUID`   | Assigned by Maskor on first detection if missing. Never changed after assignment.           |
| `title`         | `title: string`        | Display name.                                                                               |
| `version`       | `version: number`      | Incremented by Maskor on each sync. User should not edit.                                   |
| `pool`          | `pool: Pool`           | One of `unprocessed`, `incomplete`, `unplaced`, `discarded`. User may set manually.         |
| `readyStatus`   | `readyStatus: number`  | Float `0–1`. User-set, but can be auto-generated by Maskor. Source of truth is always file. |
| `notes`         | `notes: string[]`      | List of note titles (unique). Maskor appends when a note references this fragment.          |
| `references`    | `references: string[]` | List of reference names (unique). Same rules as notes.                                      |

### Inline aspect fields (markdown-owned)

Aspects are stored as inline fields below the frontmatter, using Dataview-compatible syntax. The aspect name must be unique within the project. The value is the weight (`0–1`), user-set or auto-generated by Maskor. Source of truth is always the file.

```
aspect-name:: 0.8
other-aspect:: 0.3
```

The watcher parses these lines, resolves the aspect name to a UUID in the DB, and updates `FragmentProperties` accordingly. If an aspect name is not found in the DB, Maskor logs a warning but does not fail.

Maps to `properties: FragmentProperties` on `Fragment`, where each entry is `{ aspect: Aspect, weight: number }`.

### Body (markdown-owned)

Everything after the frontmatter and inline fields. Maps to `content: Markdown` on `Fragment`. Maskor never modifies the body.

### Example file

```md
---
uuid: "frag-uuid-here"
title: "Opening scene"
version: 4
pool: unplaced
readyStatus: 0.7
notes:
  - "grief note"
references:
  - "Toni Morrison interview"
---

grief:: 0.8
city:: 0.4

Fragment body text starts here...
```

### DB-only fields

Never written to the file.

| DB field           | Notes                                                                                |
| ------------------ | ------------------------------------------------------------------------------------ |
| `contentHash`      | Hash of the body at last sync. Used to detect edits.                                 |
| `updatedAt`        | Set by Maskor on sync, not by the file's mtime.                                      |
| Sequence positions | Fragment index within any `Sequence` / `Section`. All sequence data lives in the DB. |
| Fitting scores     | Computed from aspects, arcs and sequence context.                                    |
| Arc positions      | Where the fragment sits on each arc at its current sequence position.                |

## Aspects

**File:** `aspects/<key>.md`

Aspect definitions live in the vault. The user creates and manages aspects here. Maskor assigns a UUID on first detection.

### Frontmatter (markdown-owned)

Maps to fields on the `Aspect` type.

| Frontmatter key | Type field           | Notes                                             |
| --------------- | -------------------- | ------------------------------------------------- |
| `uuid`          | `uuid: AspectUUID`   | Assigned by Maskor on first detection if missing. |
| `key`           | `key: string`        | Unique name. Used in fragment inline fields.      |
| `category`      | `category?: string`  | Optional group label (e.g. `"theme"`, `"character"`, `"setting"`). |
| `notes`         | `notes: string[]`    | List of note titles attached to this aspect.      |

### Body (markdown-owned)

Maps to `description?: string` on `Aspect`. Optional — the user may leave the body empty.

---

## Notes

**File:** `notes/<title>.md`

Notes can be created and deleted by the user freely. Maskor assigns a UUID on first detection. Note titles must be unique within the vault.

The **fragment** (or aspect) is the owner of the relationship. Its frontmatter `notes` array lists which note titles are attached. Notes themselves carry no back-reference.

### Frontmatter (markdown-owned)

Maps to fields on the `Note` type.

| Frontmatter key | Type field       | Notes                                             |
| --------------- | ---------------- | ------------------------------------------------- |
| `uuid`          | `uuid: NoteUUID` | Assigned by Maskor on first detection if missing. |
| `title`         | `title: string`  | Display name. Must be unique.                     |

### Body (markdown-owned)

Maps to `content: Markdown` on `Note`. Maskor never modifies the body.

---

## References

**File:** `references/<name>.md`

Same rules as Notes. The fragment owns the relationship via its `references` array. Reference files carry no back-reference.

### Frontmatter (markdown-owned)

Maps to fields on the `Reference` type.

| Frontmatter key | Type field            | Notes                                             |
| --------------- | --------------------- | ------------------------------------------------- |
| `uuid`          | `uuid: ReferenceUUID` | Assigned by Maskor on first detection if missing. |
| `name`          | `name: string`        | Display name. Must be unique.                     |

### Body (markdown-owned)

Maps to `content` on `Reference`. Maskor never modifies the body.

---

## Sequences and Sections

All `Sequence` and `Section` data (including fragment positions) is DB-only. No frontmatter representation. The user manages sequences exclusively through the Maskor UI.

---

## Sync Rules

**File created outside Maskor**

- Watcher detects the new file and creates a DB record.
- If `uuid` is missing from frontmatter, Maskor assigns one and writes it back.
- If `uuid` collides with an existing record, Maskor assigns a new one, writes it back, and logs a warning.

**File edited outside Maskor**

- Watcher detects the change via `contentHash` comparison and filesystem events.
- Frontmatter and inline fields are re-synced to the DB.
- DB-only fields (scores, positions) are recalculated as needed.
- Unknown aspect names in inline fields produce a warning, not a hard error.

**File renamed**

- Entities are tracked by `uuid`, not filename.
- A rename updates the DB path record only.
- Any entity that references this file by title (e.g. a note's `attachedTo`) is updated in the DB. Maskor does not rewrite other files' frontmatter on rename — this may cause stale title references in the vault until those files are next synced.

**File deleted**

- Fragment is moved to `discarded` pool, not hard-deleted from the DB.
- Notes and references are soft-deleted (marked inactive in DB). No automatic hard deletion.

**Conflict (Maskor and user edit frontmatter concurrently)**

- Last-write-wins for most fields.
- `pool` conflicts: Maskor defers to the file and logs the conflict.
- `version`: Maskor always increments on write; a stale version in the file triggers a warning but not a hard error.

**DB lost or corrupted**

- Run a full rebuild scan over the vault. All markdown-owned properties are recovered exactly.
- DB-only properties (sequence positions, fitting scores, arc positions) are lost and must be recalculated or re-entered.

## Open Questions

- **Name uniqueness enforcement** — titles/names must be unique, but this is hard to enforce when the user edits outside Maskor. Define the conflict resolution strategy (e.g. first-seen wins, last-modified wins, manual resolution required).
- **Stale note/reference titles on rename** — when a note or reference is renamed, the fragment's `notes`/`references` frontmatter arrays may contain the old title. Decide whether Maskor should rewrite fragment files proactively or resolve stale references lazily on next sync.
- **`version` field** — is a user-visible version in frontmatter useful, or should it be DB-only?
- **Auto-generated `weight` / `readyStatus` write-back** — define the trigger: on save, on explicit user action, or on sequencer run?

# Sync Contract

Defines the boundary between **markdown files** (user-owned) and the **database** (Maskor-owned).

The DB is a cache and index derived from vault files — always fully rebuildable. Markdown is source of truth for everything it owns.

---

## Vault Structure

```
<vault>/
  fragments/            # Fragment files
  fragments/discarded/  # Discarded fragments
  aspects/              # Aspect definitions
  notes/                # Note files
  references/           # Reference files
  pieces/               # Consume directory — raw pieces dropped here are converted to fragments
```

---

## Fragments

**File:** `fragments/<title>.md`

Each fragment file has a YAML frontmatter block, optional inline aspect fields, and a markdown body.

Discarded fragments are moved to `fragments/discarded/`. This overrides the `pool` property, but `pool` should automatically update. Moving a discarded fragment out of `discarded/` should automatically update `pool`.

### Frontmatter (markdown-owned)

| Frontmatter key | Type field             | Notes                                                                                       |
| --------------- | ---------------------- | ------------------------------------------------------------------------------------------- |
| `uuid`          | `uuid: FragmentUUID`   | Assigned by Maskor on first detection if missing. Never changed after assignment.           |
| `title`         | `title: string`        | Display name.                                                                               |
| `version`       | `version: number`      | Incremented by Maskor on each sync. User should not edit.                                   |
| `pool`          | `pool: Pool`           | One of `unprocessed`, `incomplete`, `unplaced`, `discarded`. User may set manually.         |
| `readyStatus`   | `readyStatus: number`  | Float `0–1`. User-set, but can be auto-generated by Maskor. Source of truth is always file. |
| `notes`         | `notes: string[]`      | List of note titles (unique). Maskor appends when a note references this fragment.          |
| `references`    | `references: string[]` | List of reference names (unique). Same rules as notes.                                      |

### Inline aspect fields (markdown-owned)

Stored as Dataview-compatible inline fields below the frontmatter. Aspect name must be unique within the project. Value is weight (`0–1`), user-set or auto-generated.

```
aspect-name:: 0.8
other-aspect:: 0.3
```

Rebuild parses these lines and resolves aspect name to UUID via `aspectsTable`. Both raw key and resolved UUID are stored in `fragment_properties` — UUID column is `NULL` if no active aspect matches. Unresolved keys produce `SyncWarning { kind: "UNKNOWN_ASPECT_KEY" }` from `rebuild()` (and from watcher on sync). Maskor never auto-modifies fragment files — user is prompted to resolve mismatches.

Maps to `properties: FragmentProperties` on `Fragment`, where each entry is `{ aspect: Aspect, weight: number }`.

### Body (markdown-owned)

Everything after frontmatter and inline fields. Maps to `content: Markdown` on `Fragment`. Maskor never modifies the body.

### Example file

```md
---
uuid: "frag-uuid-here"
title: "Opening scene"
version: 4
pool: unplaced
readyStatus: 0.7
notes:
  - "grief note"
references:
  - "Toni Morrison interview"
---

grief:: 0.8
city:: 0.4

Fragment body text starts here...
```

### DB-only fields

| DB field           | Notes                                                                                |
| ------------------ | ------------------------------------------------------------------------------------ |
| `contentHash`      | Hash of the body at last sync. Used to detect edits.                                 |
| `updatedAt`        | Set by Maskor on sync, not by the file's mtime.                                      |
| Sequence positions | Fragment index within any `Sequence` / `Section`. All sequence data lives in the DB. |
| Fitting scores     | Computed from aspects, arcs and sequence context.                                    |
| Arc positions      | Where the fragment sits on each arc at its current sequence position.                |

## Aspects

**File:** `aspects/<key>.md`

User creates and manages aspect definitions here. Maskor assigns a UUID on first detection.

### Frontmatter (markdown-owned)

| Frontmatter key | Type field          | Notes                                                              |
| --------------- | ------------------- | ------------------------------------------------------------------ |
| `uuid`          | `uuid: AspectUUID`  | Assigned by Maskor on first detection if missing.                  |
| `key`           | `key: string`       | Unique name. Used in fragment inline fields.                       |
| `category`      | `category?: string` | Optional group label (e.g. `"theme"`, `"character"`, `"setting"`). |
| `notes`         | `notes: string[]`   | List of note titles attached to this aspect.                       |

### Body (markdown-owned)

Maps to `description?: string` on `Aspect`. Optional.

---

## Notes

**File:** `notes/<title>.md`

User creates and deletes freely. Maskor assigns UUID on first detection. Titles must be unique within the vault. The fragment (or aspect) owns the relationship — its `notes` array lists attached note titles. Notes carry no back-reference.

### Frontmatter (markdown-owned)

| Frontmatter key | Type field       | Notes                                             |
| --------------- | ---------------- | ------------------------------------------------- |
| `uuid`          | `uuid: NoteUUID` | Assigned by Maskor on first detection if missing. |
| `title`         | `title: string`  | Display name. Must be unique.                     |

### Body (markdown-owned)

Maps to `content: Markdown` on `Note`. Maskor never modifies the body.

---

## References

**File:** `references/<name>.md`

Same rules as Notes. Fragment owns the relationship via its `references` array. Reference files carry no back-reference.

### Frontmatter (markdown-owned)

| Frontmatter key | Type field            | Notes                                             |
| --------------- | --------------------- | ------------------------------------------------- |
| `uuid`          | `uuid: ReferenceUUID` | Assigned by Maskor on first detection if missing. |
| `name`          | `name: string`        | Display name. Must be unique.                     |

### Body (markdown-owned)

Maps to `content` on `Reference`. Maskor never modifies the body.

---

## Sequences and Sections

All `Sequence` and `Section` data (including fragment positions) is DB-only. No frontmatter representation. User manages sequences exclusively through the Maskor UI.

---

## Sync Rules

**File created outside Maskor**

- Watcher creates a DB record.
- If `uuid` missing, Maskor assigns one and writes it back.
- If `uuid` collides, Maskor assigns a new one, writes it back, logs a warning.

**File edited outside Maskor**

- Watcher detects change via `contentHash` comparison and filesystem events.
- Frontmatter and inline fields re-synced to DB.
- DB-only fields (scores, positions) recalculated as needed.
- Unknown aspect names produce a warning, not a hard error.

**File renamed**

- Entities tracked by `uuid`, not filename — a rename updates only the DB path record.
- Title references in other entities' frontmatter are updated in DB. Maskor does not rewrite other files' frontmatter — stale title references may persist until those files are next synced.

**File deleted**

- Fragment moved to `discarded` pool, not hard-deleted.
- Notes and references soft-deleted in DB. No automatic hard deletion.

**Conflict (Maskor and user edit frontmatter concurrently)**

- Last-write-wins for most fields.
- `pool` conflict: Maskor defers to file, logs conflict.
- `version`: Maskor always increments on write; stale version in file triggers a warning, not a hard error.

**DB lost or corrupted**

- Full rebuild recovers all markdown-owned properties.
- DB-only properties (sequence positions, fitting scores, arc positions) are lost — must be recalculated or re-entered.

## Open Questions

- **Name uniqueness enforcement** — titles/names must be unique but hard to enforce on external edits. Define conflict resolution (first-seen wins, last-modified wins, manual resolution required).
- **Stale note/reference titles on rename** — when a note/reference is renamed, fragment frontmatter may contain the old title. Decide: proactive rewrite of fragment files, or lazy resolution on next sync.
- **`version` field** — is a user-visible version in frontmatter useful, or should it be DB-only?
- **Auto-generated `weight` / `readyStatus` write-back** — define the trigger: on save, on explicit user action, or on sequencer run?

# Sync Contract

Defines the boundary between **markdown files** (user-owned) and the **database** (Maskor-owned).

The DB is a cache and index derived from vault files â€” always fully rebuildable. Markdown is source of truth for everything it owns.

---

## Vault Structure

```
<vault>/
  fragments/            # Fragment files
  fragments/discarded/  # Discarded fragments
  aspects/              # Aspect definitions
  notes/                # Note files
  references/           # Reference files
  pieces/               # Consume directory â€” raw pieces dropped here are converted to fragments
```

---

## Fragments

**File:** `fragments/<title>.md`

Each fragment file has a YAML frontmatter block, optional inline aspect fields, and a markdown body.

Discarded fragments are moved to `fragments/discarded/`.

### Frontmatter (markdown-owned)

| Frontmatter key | Type field             | Notes                                                                                       |
| --------------- | ---------------------- | ------------------------------------------------------------------------------------------- |
| `uuid`          | `uuid: string`         | Assigned by Maskor on first detection if missing. Never changed after assignment.           |
| `title`         | `title: string`        | Display name.                                                                               |
| `updatedAt`     | `updatedAt: string`    | ISO 8601 timestamp. Written by Maskor on every API write. User should not edit.             |
| `readyStatus`   | `readyStatus: number`  | Float `0â€“1`. User-set, but can be auto-generated by Maskor. Source of truth is always file. |
| `notes`         | `notes: string[]`      | List of note titles (unique). Maskor appends when a note references this fragment.          |
| `references`    | `references: string[]` | List of reference names (unique). Same rules as notes.                                      |

### Inline aspect fields (markdown-owned)

Stored as Dataview-compatible inline fields below the frontmatter. Aspect name must be unique within the project. Value is weight (`0â€“1`), user-set or auto-generated.

```
aspect-name:: 0.8
other-aspect:: 0.3
```

Rebuild parses these lines and detects drift via a `Set<aspectKey>` built from the aspects pass. Unresolved keys produce `SyncWarning { kind: "UNKNOWN_ASPECT_KEY" }` from `rebuild()` (and from watcher on sync). Maskor never auto-modifies fragment files â€” user is prompted to resolve mismatches.

`fragment_properties` stores `(fragment_uuid, aspect_key, weight)` â€” no `aspect_uuid` column. Aspect names are unique within a vault, so `aspect_key` is sufficient as the join column.

Maps to `properties: FragmentProperties` on `Fragment`, where each entry is `{ weight: number }`.

### Body (markdown-owned)

Everything after frontmatter and inline fields. Maps to `content: Markdown` on `Fragment`. Maskor never modifies the body.

### Example file

```md
---
uuid: "frag-uuid-here"
title: "Opening scene"
updatedAt: "2026-04-23T10:00:00.000Z"
readyStatus: 0.7
notes:
  - "grief note"
references:
  - "Toni Morrison interview"
---

grief:: 0.8
city:: 0.4

Fragment body text starts here...
```

### DB-only fields

| DB field           | Notes                                                                                |
| ------------------ | ------------------------------------------------------------------------------------ |
| `contentHash`      | Hash of the full file (frontmatter + body) at last sync. Used for change detection. |
| `syncedAt`         | Unix ms; set by Maskor on each DB sync event.                                        |
| `filePath`         | Relative to vault root; updated on file rename.                                      |
| `isDiscarded`      | Derived from `filePath.startsWith("discarded/")` at index time.                     |
| Sequence positions | Fragment index within any `Sequence` / `Section`. All sequence data lives in the DB. |
| Fitting scores     | Computed from aspects, arcs and sequence context.                                    |
| Arc positions      | Where the fragment sits on each arc at its current sequence position.                |

## Aspects

**File:** `aspects/<key>.md`

User creates and manages aspect definitions here. Maskor assigns a UUID on first detection.

### Frontmatter (markdown-owned)

| Frontmatter key | Type field          | Notes                                                              |
| --------------- | ------------------- | ------------------------------------------------------------------ |
| `uuid`          | `uuid: string`      | Assigned by Maskor on first detection if missing.                  |
| `key`           | `key: string`       | Unique name. Used in fragment inline fields.                       |
| `category`      | `category?: string` | Optional group label (e.g. `"theme"`, `"character"`, `"setting"`). |
| `notes`         | `notes: string[]`   | List of note titles attached to this aspect.                       |

### Body (markdown-owned)

Maps to `description?: string` on `Aspect`. Optional.

---

## Notes

**File:** `notes/<title>.md`

User creates and deletes freely. Maskor assigns UUID on first detection. Titles must be unique within the vault. The fragment (or aspect) owns the relationship â€” its `notes` array lists attached note titles. Notes carry no back-reference.

### Frontmatter (markdown-owned)

| Frontmatter key | Type field       | Notes                                             |
| --------------- | ---------------- | ------------------------------------------------- |
| `uuid`          | `uuid: NoteUUID` | Assigned by Maskor on first detection if missing. |
| `title`         | `title: string`  | Display name. Must be unique.                     |

### Body (markdown-owned)

Maps to `content: Markdown` on `Note`. Maskor never modifies the body.

---

## References

**File:** `references/<name>.md`

Same rules as Notes. Fragment owns the relationship via its `references` array. Reference files carry no back-reference.

### Frontmatter (markdown-owned)

| Frontmatter key | Type field            | Notes                                             |
| --------------- | --------------------- | ------------------------------------------------- |
| `uuid`          | `uuid: ReferenceUUID` | Assigned by Maskor on first detection if missing. |
| `name`          | `name: string`        | Display name. Must be unique.                     |

### Body (markdown-owned)

Maps to `content` on `Reference`. Maskor never modifies the body.

---

## Sequences and Sections

All `Sequence` and `Section` data (including fragment positions) is DB-only. No frontmatter representation. User manages sequences exclusively through the Maskor UI.

---

## Sync Rules

**File created outside Maskor**

- Watcher creates a DB record.
- If `uuid` missing, Maskor assigns one and writes it back.
- If `uuid` collides, Maskor assigns a new one, writes it back, logs a warning.

**File edited outside Maskor**

- Watcher detects change via `contentHash` comparison and filesystem events.
- Frontmatter and inline fields re-synced to DB.
- DB-only fields (scores, positions) recalculated as needed.
- Unknown aspect names produce a warning, not a hard error.

**File renamed**

- Entities tracked by `uuid`, not filename â€” a rename updates only the DB path record.
- Title references in other entities' frontmatter are updated in DB. Maskor does not rewrite other files' frontmatter â€” stale title references may persist until those files are next synced.

**File deleted**

- Fragment moved to `discarded` folder, not hard-deleted.
- Notes and references soft-deleted in DB. No automatic hard deletion.

**Conflict (Maskor and user edit frontmatter concurrently)**

- Last-write-wins for most fields.

**DB lost or corrupted**

- Full rebuild recovers all markdown-owned properties.
- DB-only properties (sequence positions, fitting scores, arc positions) are lost â€” must be recalculated or re-entered.

## Open Questions

- **Name uniqueness enforcement** â€” titles/names must be unique but hard to enforce on external edits. Define conflict resolution (first-seen wins, last-modified wins, manual resolution required).
- **Stale note/reference titles on rename** â€” when a note/reference is renamed, fragment frontmatter may contain the old title. Decide: proactive rewrite of fragment files, or lazy resolution on next sync.
- **`updatedAt` for externally-edited files** â€” when Obsidian edits a fragment directly, `updatedAt` in frontmatter is not updated by Maskor (only API writes update it). Decision pending: accept stale `updatedAt` for external edits, or have Maskor write it back on every sync.
- **Auto-generated `weight` / `readyStatus` write-back** â€” define the trigger: on save, on explicit user action, or on sequencer run?

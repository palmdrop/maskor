# Suggestions

- 2026-06-13 — **Fragment split is not atomic across its many writes (mitigated, not eliminated).** `splitFragmentCommand` performs N fragment writes + M sequence writes + up to N margin writes, each taking the vault write-lock on its own — there is no single transaction spanning the operation. The data-loss window flagged in review (`references/reviews/fragment-split-2026-06-13.md` #1) is fixed by ordering the original's truncation **last**, after every new piece is persisted, so a mid-operation failure leaves the source prose intact (worst case: a few orphan fragments + no `fragment:split` log entry, since `executeCommand` records `command:error` on throw). A genuinely atomic split would need a vault-wide transaction primitive (create + place + margin-migrate + truncate as one unit) that does not exist today. Low priority while keys are deterministically unique (so `KEY_CONFLICT` mid-loop is implausible) and there are no live users; revisit if partial-split states are observed.

- 2026-06-11 — **Fragments have no `createdAt` (blocks a "Created at" sort).** The domain `Fragment` (`packages/shared/src/schemas/domain/fragment.ts`) and the index layer (`IndexedFragment`) carry only `updatedAt`. The FragmentListPage sort dropdown therefore ships with Name + Updated at + per-sequence order, but **Created at is deferred**. Adding it means: extend the domain schema + index with `createdAt`, source it reliably (file birthtime via `fs.stat` `birthtime`, or a managed frontmatter key — markdown-as-source-of-truth makes filesystem birthtime fragile across vault copies/syncs), thread it through the API response schemas, regenerate OpenAPI + orval, then add the dropdown option. Frontmatter is the durable source; birthtime is the cheap one. Decide before implementing.

- 2026-06-02 — **Margin panel: pixel-perfect block↔comment alignment deferred.** Phase 6 ships the side-by-side Margin panel with comments ordered to follow fragment block order, collapse/expand toggles, and click-to-reveal scroll correspondence. The spec's "expanded: alignment padding so each block and its comments sit vertically beside each other" (measuring block offsets in the editor and padding the shorter column) is **not** implemented — it requires measuring rendered block geometry in both TipTap and CM6 and reconciling with the comment list, which is a large, fragile piece of layout work. The current ordering + reveal affordance covers the correspondence need pragmatically. Revisit if the alignment padding proves necessary in use. (plan: margins.md Phase 6)

- **Create command palette entries should open dialogs directly** (`packages/frontend/src/lib/commands/catalog/useProjectShellCommands.ts`): `Create fragment…`, `Create note…`, `Create reference…`, `Create aspect…` currently navigate to the creation page rather than opening the creation dialog. To open the dialog directly, lift the dialog open state out of `CreateEntityDialog` / `AttachableEntityPanel` so commands can call `setOpen(true)`. Alternatively, move the create commands into the respective page components (FragmentListPage, ProjectConfigPage tabs) with `scope: "global"` — they'd only be discoverable when on those pages, which limits the UX.

- **Backend session cache for import preview** (`packages/api/src/routes/import-preview.ts`): The preview endpoint re-uploads and re-converts the file on every options change (heading level / delimiter tweak). For larger `.docx` files this is wasteful — mammoth + turndown runs on every debounced call. Consider a backend session cache keyed by content hash (SHA-256 of file bytes) with a short TTL (e.g. 10 min) that stores the converted markdown so subsequent preview requests with different options skip the mammoth+turndown step entirely. Trigger this optimization when preview latency becomes user-visible (measure first).

- **Import Pipeline PRD has contradictory empty-piece ACs** (`tasks/prd-import-pipeline-stage-1.md`): US-001 says empty pieces "are not emitted by the splitter; the importCommand records them in errors[]" — those two clauses contradict. US-004 then explicitly requires reporting them in errors[]. Current implementation honors clause 1 (splitters filter, importCommand has no empty path). To honor the API contract literally, splitters would need to emit empties (or return a count) so importCommand can report them. Either tighten splitter contract or relax US-004. Flagged in `references/reviews/import-pipeline-stage-1-2026-05-16.md` (issue #3).

- **Add GFM table support to TipTap editor** (`packages/frontend/src/components/prose-editor.tsx`): Tables are not in scope for the initial TipTap implementation. Install `@tiptap/extension-table` and add to the extensions array if fragments start containing markdown tables.

- **`Vim.defineEx("w", ...)` is global and persists after unmount** (`packages/frontend/src/components/prose-editor.tsx`): Calling `Vim.defineEx` in `onCreateEditor` registers the command globally on the vim state machine — it won't be cleaned up when the component unmounts. In practice this is harmless (no editor = no `:w`), but if multiple `ProseEditor` instances exist simultaneously they'd overwrite each other's `:w` binding. Revisit if supporting multi-pane editing.

- **Add MSW for frontend API mocking** (`packages/frontend/src/test/`): The testing framework is set up but MSW (Mock Service Worker) is not yet installed. Once the frontend starts making API calls, add `msw` and create handler fixtures in `src/test/handlers.ts` so tests intercept HTTP at the network level rather than mocking `fetch` manually. Install: `bun add -D msw --cwd packages/frontend`.

- **Port/environment config** (`packages/api/src/index.ts`, `packages/frontend/`): API defaults to `3001` via `process.env.PORT`. When wiring the frontend, ensure both agree on port — consider a shared `.env` file at the repo root or a workspace-level config package to avoid silent mismatches.

- **`vaultPath` discovery for frontend** (`packages/api/src/routes/projects.ts`): `POST /projects` expects a raw filesystem path in the body. The frontend (Tauri/Electron) must know the path to supply it. Deferred options: expose a file-picker over IPC, add a `/projects/discover` endpoint, or drive from a config file. Resolve before frontend integration begins.

- **`rebuild()` loads all vault entities into memory** (`packages/storage/src/index/indexer.ts`): The two-phase approach (read all → write in transaction) holds all fragment/aspect/note/reference entries in memory simultaneously. For very large vaults this may become a concern. Consider chunked or streaming writes as a follow-up if memory pressure is observed.

- **Bulk / auto-discovery recovery for registry loss** (`packages/storage/src/registry/registry.ts`, `packages/frontend/src/pages/ProjectManagementPage/`): Adopt mode now reuses manifest UUIDs (`registry.ts:134`), so the manual recovery primitive exists — re-adopt each vault folder and UUIDs + per-project config are preserved. What's still missing is a bulk path: a `recoverFromManifests(vaultRoots: string[])` function that walks the maskor-managed root (and any other configured roots), reads `.maskor/project.json` manifests, and re-inserts missing rows in one shot, plus a UI surface on the project page ("Found in your Maskor folder, not registered — Adopt all / Adopt individually"). Don't auto-adopt — surface only. Cache the scan result per app session. Useful for: registry DB loss, restoring from backup, switching machines. Both the bulk primitive and the auto-discovery UI were explicit Non-Goals of the project-management PRD; revisit if registry-loss reports surface or backup/restore becomes a real flow.

- **Strip hardcoded `rounded-*` from shadcn components** (`packages/frontend/src/components/ui/`): shadcn-generated components may contain hardcoded `rounded-md` / `rounded-sm` classes that survive the `--radius: 0rem` CSS variable override. Audit the generated files and remove any `rounded-*` utilities that clash with the flat editorial theme.

- **`TagCombobox` has no keyboard shortcut to dismiss without selecting** (`packages/frontend/src/components/ui/tag-combobox.tsx`): Pressing `Escape` closes the popover and blurs the input, but there is no way to keep focus in the input while dismissing the dropdown. If the UX calls for it, add a second `Escape` press that re-focuses without reopening.

- **Raw async functions from orval are PascalCase** (`packages/frontend/src/api/generated/`): The `operationName` capitalizer makes type names PascalCase but also affects the raw async functions (`ListFragments`, `GetProject`, etc.). These are unconventional for JS variables. If it becomes a problem, look into a custom `transformer` in orval config that renames only the type aliases post-generation while keeping function names camelCase.

- **Watcher start is a side effect in `resolveProject` middleware** (`packages/api/src/middleware/resolve-project.ts`): Starting the watcher inside request middleware is pragmatic but mixes concerns — a side effect (process-level resource acquisition) is hidden inside what reads as a lookup. When project lifecycle management is formalised (explicit project open/close flow), move watcher start to an explicit startup call rather than a middleware side effect.

- **Consider stale-entry approach for external rename detection** (`packages/storage/src/watcher/watcher.ts`): The current plan uses a 500ms timer buffer to correlate `unlink`+`add` events. An alternative: on unlink, mark the DB entry as stale (add `deletedAt`) instead of deleting it. On `add`, match by UUID (rename) or key (replacement) and overwrite the stale row. Benefits: no timing dependency, handles the "temporarily moved out of vault" case natively, deterministic collision resolution. Costs: schema change on notes/references tables, all read queries must filter stale rows, stale entries accumulate for true deletions (needs periodic cleanup). See `references/plans/external-rename-cascade.md`.

- **Rebuild mutex has an async race window** (`packages/storage/src/watcher/watcher.ts` `pause()`): `isPaused = true` is set synchronously, but any handler already past the `if (isPaused) return` check and mid-await will still complete and upsert into the DB after `pause()` returns. Rebuild's transaction then overwrites that upsert — the exact race the mutex was designed to prevent. A full fix requires draining in-flight handlers before rebuild begins (e.g. a counter of active handlers with a promise that resolves when it reaches zero). See review: `references/reviews/storage-sync-spec-fixes-2026-04-23.md` (warning #4).

- **`PRAGMA foreign_keys` was off on the vault DB** (`packages/storage/src/db/vault/index.ts`): Foreign key enforcement was silently disabled on the vault SQLite database, meaning `ON DELETE CASCADE` rules on `sections` and `fragment_positions` were not firing. Enabled in Phase 4 of the sequencer slice. Any pre-existing code that relied on cascades (e.g. deleting a sequence row and expecting its sections to vanish) was silently broken before this fix. Verify no other DB creation sites are missing the pragma.

- **dnd-kit codegen conflicts**: `@dnd-kit/sortable@10` ships with `arrayMove` but `@dnd-kit/utilities` does not re-export it. Import `arrayMove` from `@dnd-kit/sortable`, not `@dnd-kit/utilities`.

- **`upsertSequence` does not hash-skip during rebuild** (`packages/storage/src/indexer/upserts.ts`): Unlike fragment/aspect upserts which could skip unchanged entries via hash comparison, `upsertSequence` always deletes and re-inserts sections and fragment_positions. This is fine for the current single-sequence use case (sequences are small), but will become a concern if rebuild is called frequently with many sequences. Consider adding a hash-guard similar to fragment upserts.

- **Test action-log query invalidation after mutations** (`packages/frontend/src/`): `useInvalidateActionLog` is called inside each editor's `onSuccess` mutation handler — currently untested. A proper test needs MSW to intercept the PATCH request and a full React Query integration setup to assert the action log query key is invalidated and re-fetched. Depends on MSW being set up first (see the existing MSW suggestion above). Low priority: the invalidation path is a single `queryClient.invalidateQueries` call with no branching logic, so the risk of it being wrong is low.

- **Resilient action-log append failures** (`packages/api/src/commands/`, `packages/storage/src/action-log/`): v1 catches log-append failures and emits an `error`-level structured log only — entries that failed to write are lost from the canonical log file. The spec (`specifications/action-log.md` § Failure handling) describes the intended behavior: write the failed entry to a sidecar `<vault>/.maskor/action-log.failed.jsonl` so it is preserved and inspectable, and surface a low-key banner on the history page when failed entries exist in the session. Implement once v1 is shipped and the writer is stable. Adds two pieces: a sidecar writer (mirrors the main writer interface, simpler — no rotation needed) and a session-level "failed-append count" exposed via a small endpoint or via the existing health surface, plus a banner component on the history page.

- **Sequence-name uniqueness is case-sensitive; fragment-key uniqueness is case-insensitive** (`packages/storage/src/service/storage-service.ts`): US-004 (secondary sequences PRD) specified case-sensitive sequence-name collision detection, so the storage layer now allows "Main" and "main" to coexist in the same project. Fragment keys, by contrast, still use case-insensitive comparison. This split was intentional per the PRD but is a UX wart: two sequences differing only by case look identical in a sidebar list. Consider switching sequence-name comparison to case-insensitive (drop the case-sensitive flavor) for consistency with fragment keys and clearer UX. Would need PRD/spec amendment and a one-line code change.

- **Per-item list actions deferred to Phase 5 (parameterized commands)** (`packages/frontend/src/pages/`): Three groups of per-fragment/per-entity actions were intentionally not migrated to the command system in Phase 2 because they require dynamic IDs or parameterized dispatch: (1) `FragmentListPage` — Restore, Delete, Discard buttons per fragment row; (2) `NotesTab`/`ReferencesTab` per-item delete via `AttachableEntityPanel`; (3) `ArcEditor` — Save arc per aspect (multiple simultaneously mounted). All three need `CommandArg` / two-step parameterized commands (Phase 5) before they can be palette-discoverable.

- **Draft preview/overview** (deferred from `specifications/drafting.md`): v1 ships drafts as create + list + restore only — no in-app inspection of a draft's contents beyond name/date/note metadata. Two future paths, not mutually exclusive: (1) Once a "linear preview" mode exists for the main sequence (full readable text top-to-bottom), point it at any draft's snapshot DB / vault files so the user can read an old draft end-to-end without restoring. (2) At draft-creation time, generate a `preview.md` inside the draft directory by concatenating the main sequence's fragment content in order — gives the user a single self-contained markdown file to scan or share, even outside Maskor. Option 2 is cheap and useful immediately; option 1 is the proper in-app inspector. Decide which when the linear preview mode lands.

- **OpenAPI path params must use `{name}` syntax, not `:name`** (`packages/api/src/routes/*`): `createRoute({ path: "/visit/:fragmentId" })` parses for Hono routing but emits a literal `:fragmentId` into the OpenAPI document, which then ends up in the generated orval URL builder as a literal string. The fragment-visit and (originally) fragment-pick routes both hit this — both fixed 2026-05-26. Convention: always use `path: "/visit/{fragmentId}"` for any param besides the project-id. A lint rule or codegen verification step on the OpenAPI document would catch this in the future.

- **Novel-scale rendering of the preview in a single read-only Tiptap instance** (`packages/frontend/src/pages/PreviewPage/`, planned shared read-only renderer): The preview/import rendering refactor renders the whole assembled document in ONE read-only Tiptap/ProseMirror instance, replacing the per-fragment `StaticMarkdown`. One instance removes the per-fragment editor explosion the original spec feared — but ProseMirror does not virtualize, so an entire novel-sized document (100k+ words) lives in the DOM at once. Validate rendering (not just assembly) at 100k+ words during the build. If the live instance stutters, render the document to static HTML from the _same_ Tiptap schema (`generateHTML(doc, sharedExtensions)`) instead of a live editor: identical typography, anchors come through via the anchor node's `renderHTML`, and the HTML is safe to inject because it is schema-generated, not raw user input. Treat as a measured optimization — only switch if the live instance proves too heavy. Deferred per grill on 2026-05-30.

- **Spec overstated "DB is fully re-derivable from the vault"** (`specifications/storage-sync.md`, `packages/storage/src/db/vault/schema.ts`): The invariant is not strictly true. `fragment_stats` carries behavioral telemetry (`voluntaryOpenCount`, `promptAcceptCount`, `avoidanceCount`, `editCount`, `lastSurfacedAt`) accumulated at runtime by the prompting mechanism and stored in **no** vault file — `index.rebuild` only backfills `wordCount`/`excerpt` and preserves the rest via `onConflictDoNothing`. `UUID_COLLISION` event warnings are likewise never re-derived. So a normal rebuild keeps both, but any full DB drop (e.g. the dev auto-reset, `references/plans/dev-db-auto-reset.md`) discards them. Spec wording corrected 2026-06-01 to qualify the invariant. Longer term, if prompting telemetry becomes valuable to preserve, it needs a vault-backed home (or an explicit "this is canonical DB-only state" carve-out) — otherwise it is silently the one thing a vault can't reconstruct.

- **Overview reorder list and prose spine are separate DnD contexts — no cross-surface drag** (`packages/frontend/src/pages/OverviewPage/index.tsx`, `ReorderList.tsx`, `ProseSpine.tsx`): The overview-redesign (2026-06-05) makes both the left reorder list and the main prose spine draggable, but they run in two independent `DndContext`s because both use raw fragment uuids as draggable ids and dnd-kit requires ids to be unique within a single context. Consequence: you cannot drag in one gesture from the pool (which lives only in the left list) onto the spine, or from the spine back to the pool — pool place/unplace happens in the list; the spine only reorders already-placed fragments. To unify into one drag surface (pool row draggable directly onto the spine, spine fragment draggable to the pool), give each surface a prefixed draggable/droppable id namespace (e.g. `list:<uuid>` / `spine:<uuid>`, `list-section:<uuid>` / `spine-section:<uuid>`) and strip the prefix in `useSequenceDnD.handleDragEnd` before resolving sections/pool. Single context, no id collision, full cross-surface DnD. Deferred — current split is functional; revisit if users expect to drag pool→spine directly.

- **`global-create-dialogs.tsx`'s local create descriptor table has no clean home in the merged entity registry — an extension, not a fold-in** (`packages/frontend/src/components/global-create-dialogs.tsx`, `packages/frontend/src/lib/entity-kinds/{registry,entityHooks}.ts`): Phase 3 collapsed the four inline create flows into one descriptor table (title, create mutation, secondary-field config, post-create navigation). The original plan was to fold this into Plan 1's per-kind registry once it landed. Plan 1 has now merged — but on inspection its registry is **edit/extract/insert**-only: `ENTITY_HOOKS` carries GET + UPDATE hooks, `ENTITY_KIND_META` carries extract/insert metadata; **neither has a create hook or a navigation route**. So there is no create slot to fold into — co-locating would mean _extending_ the registry with a new `ENTITY_CREATE` hooks map (with the same `as unknown` casts `ENTITY_HOOKS` uses) plus per-kind navigation. Navigation is the sticking point: TanStack's typed router rejects a dynamic `to` string, so the routes can't be plain registry data — they have to stay as per-kind closures (`(navigate, projectId, uuid) => void`). Net: it's a relocation that co-locates "how to add an entity kind" (a real readability win) at the cost of more cast-heavy plumbing in the shared module; it does **not** remove duplication (create vs. edit are different per-kind concerns that merely share the `EntityKind` key, which `ENTITY_KINDS` already centralizes). **Worth doing only if you want a single "add a kind here" surface; otherwise the self-contained, tested local table is fine.** Decision deferred to the developer.

Backend now imports the logger from the explicit `@maskor/shared/logger` subpath (the one `createLogger` value import plus ~19 `import type { Logger }` sites were updated). The frontend's hard `@maskor/shared` aliases (in `vitest.config.ts` and `tsconfig.app.json`) were removed so resolution honors the `exports` map (`moduleResolution: "bundler"` + the workspace symlink); subpath imports like `@maskor/shared/utils` now work and never touch the logger. Verified: a frontend value import from both the barrel and `/utils` resolves with no logger leak; typecheck, backend + frontend tests, and the prod `vite build` all pass. **Follow-ups since done:** (1) `extract-utils.ts` now imports `ENTITY_KEY_REGEX` from `@maskor/shared` instead of re-declaring it; (2) the anchor-sentinel protocol was moved to `@maskor/shared/sentinel` (new browser-safe subpath) and the exporter's `sentinel.ts` + the frontend's `anchor-sentinel.ts` mirror were both deleted — exporter and frontend now share one definition. - **Backend session cache for import preview** (`packages/api/src/routes/import-preview.ts`): The preview endpoint re-uploads and re-converts the file on every options change (heading level / delimiter tweak). For larger `.docx` files this is wasteful — mammoth + turndown runs on every debounced call. Consider a backend session cache keyed by content hash (SHA-256 of file bytes) with a short TTL (e.g. 10 min) that stores the converted markdown so subsequent preview requests with different options skip the mammoth+turndown step entirely. Trigger this optimization when preview latency becomes user-visible (measure first).

- **Import Pipeline PRD has contradictory empty-piece ACs** (`tasks/prd-import-pipeline-stage-1.md`): US-001 says empty pieces "are not emitted by the splitter; the importCommand records them in errors[]" — those two clauses contradict. US-004 then explicitly requires reporting them in errors[]. Current implementation honors clause 1 (splitters filter, importCommand has no empty path). To honor the API contract literally, splitters would need to emit empties (or return a count) so importCommand can report them. Either tighten splitter contract or relax US-004. Flagged in `references/reviews/import-pipeline-stage-1-2026-05-16.md` (issue #3).

For very large vaults this may become a concern. Consider chunked or streaming writes as a follow-up if memory pressure is observed.

- **Consider partial indexes on `deleted_at IS NULL`** (`packages/storage/src/db/vault/schema.ts`): Composite and single-column indexes on `(is_discarded, deleted_at)` and `deleted_at` are now in place. If the vault grows large and most rows are active, a partial index (`WHERE deleted_at IS NULL`) would be more space-efficient — SQLite supports this via raw SQL but not yet via the Drizzle schema DSL.

- **Bulk / auto-discovery recovery for registry loss** (`packages/storage/src/registry/registry.ts`, `packages/frontend/src/pages/ProjectManagementPage/`): Adopt mode now reuses manifest UUIDs (`registry.ts:134`), so the manual recovery primitive exists — re-adopt each vault folder and UUIDs + per-project config are preserved. - **Watcher start is a side effect in `resolveProject` middleware** (`packages/api/src/middleware/resolve-project.ts`): Starting the watcher inside request middleware is pragmatic but mixes concerns — a side effect (process-level resource acquisition) is hidden inside what reads as a lookup. - **Consider stale-entry approach for external rename detection** (`packages/storage/src/watcher/watcher.ts`): The current plan uses a 500ms timer buffer to correlate `unlink`+`add` events. An alternative: on unlink, mark the DB entry as stale (add `deletedAt`) instead of deleting it. On `add`, match by UUID (rename) or key (replacement) and overwrite the stale row. Benefits: no timing dependency, handles the "temporarily moved out of vault" case natively, deterministic collision resolution. Costs: schema change on notes/references tables, all read queries must filter stale rows, stale entries accumulate for true deletions (needs periodic cleanup). See `references/plans/external-rename-cascade.md`.

- **`deletedAt` column is dead weight on notes and references** (`packages/storage/src/db/vault/schema.ts`): Notes and references were switched to hard-delete, but `notesTable` and `referencesTable` still carry a `deletedAt` column and `upsertNote`/`upsertReference` still write `deletedAt: null`. Drop the column in a migration and remove the field from the upserts. Aspects and fragments still use soft-delete, so their `deletedAt` columns remain intentional.

- **Rebuild mutex has an async race window** (`packages/storage/src/watcher/watcher.ts` `pause()`): `isPaused = true` is set synchronously, but any handler already past the `if (isPaused) return` check and mid-await will still complete and upsert into the DB after `pause()` returns. Rebuild's transaction then overwrites that upsert — the exact race the mutex was designed to prevent. A full fix requires draining in-flight handlers before rebuild begins (e.g. a counter of active handlers with a promise that resolves when it reaches zero). See review: `references/reviews/storage-sync-spec-fixes-2026-04-23.md` (warning #4).

- **`PRAGMA foreign_keys` was off on the vault DB** (`packages/storage/src/db/vault/index.ts`): Foreign key enforcement was silently disabled on the vault SQLite database, meaning `ON DELETE CASCADE` rules on `sections` and `fragment_positions` were not firing. Enabled in Phase 4 of the sequencer slice. Any pre-existing code that relied on cascades (e.g. deleting a sequence row and expecting its sections to vanish) was silently broken before this fix. Verify no other DB creation sites are missing the pragma.

- **dnd-kit codegen conflicts**: `@dnd-kit/sortable@10` ships with `arrayMove` but `@dnd-kit/utilities` does not re-export it. Import `arrayMove` from `@dnd-kit/sortable`, not `@dnd-kit/utilities`.

- **`upsertSequence` does not hash-skip during rebuild** (`packages/storage/src/indexer/upserts.ts`): Unlike fragment/aspect upserts which could skip unchanged entries via hash comparison, `upsertSequence` always deletes and re-inserts sections and fragment_positions. This is fine for the current single-sequence use case (sequences are small), but will become a concern if rebuild is called frequently with many sequences. Consider adding a hash-guard similar to fragment upserts.

- **[RESOLVED 2026-05-31] Full-suite `packages/api` test run was poisoned by watchers left running when test temp vaults were `rmSync`'d** (`packages/api/src/__tests__/helpers/create-test-app.ts`, `packages/storage/src/watcher/sync/keyed-entity.ts`, `packages/storage/src/service/storage-service.ts`): Running the whole `packages/api` directory in one `bun test` process (as `bun run verify` does) produced dozens of **non-deterministic** failures — every one a `SQLiteError: attempt to write a readonly database`. **Root cause:** `createTestApp`'s `cleanup()` only did `rmSync(temporaryDirectory)` — it never stopped the watcher that `resolveProject` middleware lazily started on the test's vault. When `rmSync` deleted the watched `.md` files, chokidar fired `unlink` events, `unlinkKeyedEntity` scheduled a 500 ms rename-buffer `setTimeout`, and ~500 ms later that timer ran `vaultDatabase.transaction(...)` against a vault whose directory no longer existed; the open `bun:sqlite` inode survived but SQLite couldn't create its journal/WAL sidecar in the deleted directory, so it reported the handle read-only. The unhandled error surfaced while a _later_ test file was running, so bun blamed whatever test was executing at that instant — hence unrelated routes (aspects/notes/sequences/suggestion/import, plus a `statfs ENOENT` in storage drafts) failing intermittently. **OS difference (why it was green on macOS, red on Linux):** chokidar uses the native backend per OS. Linux → inotify delivers per-file `unlink` events immediately, reliably scheduling the leaked timers, and the deleted-directory write fails as `SQLITE_READONLY` (can't create the journal sidecar). macOS → FSEvents coalesces a recursive subtree delete into a latent, batched event on the parent and rarely emits the per-file `unlink`, so the timers are usually never scheduled and the suite passes. A green macOS run was a **false negative** — the leak existed on every platform; only its symptom was Linux-specific, and CI (Linux) would have hit it. **Fix shipped:** `createStorageService` now exposes `shutdown()` (stops every cached watcher — `drainAll()`s rename buffers, `await`s `chokidar.close()` — and clears the cache); `createTestApp.cleanup()` is async and calls it before `rmSync`; all api test teardown hooks `await` cleanup. Full `bun run verify` went from flaky ~59 fail to **749 backend + 431 frontend pass, 0 fail, deterministic across repeated runs**.

- **Spec overstated "DB is fully re-derivable from the vault"** (`specifications/storage-sync.md`, `packages/storage/src/db/vault/schema.ts`): The invariant is not strictly true. `fragment_stats` carries behavioral telemetry (`voluntaryOpenCount`, `promptAcceptCount`, `avoidanceCount`, `editCount`, `lastSurfacedAt`) accumulated at runtime by the prompting mechanism and stored in **no** vault file — `index.rebuild` only backfills `wordCount`/`excerpt` and preserves the rest via `onConflictDoNothing`. `UUID_COLLISION` event warnings are likewise never re-derived. Longer term, if prompting telemetry becomes valuable to preserve, it needs a vault-backed home (or an explicit "this is canonical DB-only state" carve-out) — otherwise it is silently the one thing a vault can't reconstruct.

# Suggestions

- **Wire `vimMode` prop to a real settings system** (`packages/frontend/src/components/fragments/prose-editor.tsx:Props`): `vimMode` is hardcoded to `false` in `FragmentEditor`. Once a user settings/config system exists, read the vim preference from there and pass it down.
- **Add GFM table support to TipTap editor** (`packages/frontend/src/components/fragments/prose-editor.tsx`): Tables are not in scope for the initial TipTap implementation. Install `@tiptap/extension-table` and add to the extensions array if fragments start containing markdown tables.
- **`Vim.defineEx("w", ...)` is global and persists after unmount** (`packages/frontend/src/components/fragments/prose-editor.tsx`): Calling `Vim.defineEx` in `onCreateEditor` registers the command globally on the vim state machine — it won't be cleaned up when the component unmounts. In practice this is harmless (no editor = no `:w`), but if multiple `ProseEditor` instances exist simultaneously they'd overwrite each other's `:w` binding. Revisit if supporting multi-pane editing.
- **Storybook CSS imports missing from `tsconfig`** (`.storybook/preview.ts`): `tsc -b` reports `Cannot find module '../src/styles/reset.css'` in `preview.ts`. Add `"moduleResolution": "bundler"` or a `.d.ts` shim for `.css` files to the storybook tsconfig so the full build passes clean.
- **Add MSW for frontend API mocking** (`packages/frontend/src/test/`): The testing framework is set up but MSW (Mock Service Worker) is not yet installed. Once the frontend starts making API calls, add `msw` and create handler fixtures in `src/test/handlers.ts` so tests intercept HTTP at the network level rather than mocking `fetch` manually. Install: `bun add -D msw --cwd packages/frontend`.
- **`consumeAll` delete reliability** (`packages/storage/src/backend/markdown/vault.ts`): The piece file deletion in `consumeAll` calls `Bun.file().delete?.()` then falls back to `fs.unlink`. The optional chaining means a silent no-op if `delete` is absent. Consolidate to just `fs.unlink` for reliability.
- **Port/environment config** (`packages/api/src/index.ts`, `packages/frontend/`): API defaults to `3001` via `process.env.PORT`. When wiring the frontend, ensure both agree on port — consider a shared `.env` file at the repo root or a workspace-level config package to avoid silent mismatches.
- **Log level via `LOG_LEVEL` env var** (`packages/api/src/index.ts`): The logger is wired end-to-end but the level is hardcoded to `"info"`. Pass `process.env.LOG_LEVEL ?? "info"` so dev can switch to `"debug"` (vault/watcher sync details) without code changes.
- **`vaultPath` discovery for frontend** (`packages/api/src/routes/projects.ts`): `POST /projects` expects a raw filesystem path in the body. The frontend (Tauri/Electron) must know the path to supply it. Deferred options: expose a file-picker over IPC, add a `/projects/discover` endpoint, or drive from a config file. Resolve before frontend integration begins.
- **`rebuild()` loads all vault entities into memory** (`packages/storage/src/index/indexer.ts`): The two-phase approach (read all → write in transaction) holds all fragment/aspect/note/reference entries in memory simultaneously. For very large vaults this may become a concern. Consider chunked or streaming writes as a follow-up if memory pressure is observed.
- **Consider partial indexes on `deleted_at IS NULL`** (`packages/storage/src/db/vault/schema.ts`): Composite and single-column indexes on `(is_discarded, deleted_at)` and `deleted_at` are now in place. If the vault grows large and most rows are active, a partial index (`WHERE deleted_at IS NULL`) would be more space-efficient — SQLite supports this via raw SQL but not yet via the Drizzle schema DSL.
- **Implement manifest-based DB recovery** (`packages/storage/src/registry/registry.ts`): If the registry DB is lost or corrupted, the `.maskor/project.json` manifests inside each vault cannot currently be used to re-register projects. Add a `recoverFromManifests(vaultRoots: string[])` function that scans vault directories, reads manifests, and re-inserts missing rows. This makes the "self-describing vaults" promise from `SYNC_CONTRACT.md` real.
- **Add a saved OpenAPI snapshot for offline codegen** (`packages/frontend/src/api/openapi.json`): Currently `bun run codegen` requires the API to be running at `localhost:3001`. Save a snapshot with `curl http://localhost:3001/doc > src/api/openapi.json` and point `orval.config.ts` at the file for offline / CI use. Switch back to the live URL during active API development.
- **Strip hardcoded `rounded-*` from shadcn components** (`packages/frontend/src/components/ui/`): shadcn-generated components may contain hardcoded `rounded-md` / `rounded-sm` classes that survive the `--radius: 0rem` CSS variable override. Audit the generated files and remove any `rounded-*` utilities that clash with the flat editorial theme.
- **`TagCombobox` has no keyboard shortcut to dismiss without selecting** (`packages/frontend/src/components/ui/tag-combobox.tsx`): Pressing `Escape` closes the popover and blurs the input, but there is no way to keep focus in the input while dismissing the dropdown. If the UX calls for it, add a second `Escape` press that re-focuses without reopening.
- **Raw async functions from orval are PascalCase** (`packages/frontend/src/api/generated/`): The `operationName` capitalizer makes type names PascalCase but also affects the raw async functions (`ListFragments`, `GetProject`, etc.). These are unconventional for JS variables. If it becomes a problem, look into a custom `transformer` in orval config that renames only the type aliases post-generation while keeping function names camelCase.
- **`useVaultEvents` hardcodes `localhost:3001`** (`packages/frontend/src/hooks/useVaultEvents.ts:21`): The `EventSource` URL bypasses the Vite proxy, breaking in any environment where the API isn't on that exact host/port. Change to `/api/projects/${projectId}/events` so it routes through the proxy like all other API calls.
- **Watcher start is a side effect in `resolveProject` middleware** (`packages/api/src/middleware/resolve-project.ts`): Starting the watcher inside request middleware is pragmatic but mixes concerns — a side effect (process-level resource acquisition) is hidden inside what reads as a lookup. When project lifecycle management is formalised (explicit project open/close flow), move watcher start to an explicit startup call rather than a middleware side effect.
- **Consider stale-entry approach for external rename detection** (`packages/storage/src/watcher/watcher.ts`): The current plan uses a 500ms timer buffer to correlate `unlink`+`add` events. An alternative: on unlink, mark the DB entry as stale (add `deletedAt`) instead of deleting it. On `add`, match by UUID (rename) or key (replacement) and overwrite the stale row. Benefits: no timing dependency, handles the "temporarily moved out of vault" case natively, deterministic collision resolution. Costs: schema change on notes/references tables, all read queries must filter stale rows, stale entries accumulate for true deletions (needs periodic cleanup). See `references/plans/external-rename-cascade.md`.
- **`deletedAt` column is dead weight on notes and references** (`packages/storage/src/db/vault/schema.ts`): Notes and references were switched to hard-delete, but `notesTable` and `referencesTable` still carry a `deletedAt` column and `upsertNote`/`upsertReference` still write `deletedAt: null`. Drop the column in a migration and remove the field from the upserts. Aspects and fragments still use soft-delete, so their `deletedAt` columns remain intentional.
- **Rebuild mutex has an async race window** (`packages/storage/src/watcher/watcher.ts` `pause()`): `isPaused = true` is set synchronously, but any handler already past the `if (isPaused) return` check and mid-await will still complete and upsert into the DB after `pause()` returns. Rebuild's transaction then overwrites that upsert — the exact race the mutex was designed to prevent. A full fix requires draining in-flight handlers before rebuild begins (e.g. a counter of active handlers with a promise that resolves when it reaches zero). See review: `references/reviews/storage-sync-spec-fixes-2026-04-23.md` (warning #4).
- **Temporary unsaved-edit cache for entity navigation** (`packages/frontend/src/components/`): When the user navigates away from an entity editor with unsaved changes — suggestion-mode "Next", inter-entity link clicks, or any future flow that auto-saves on transition — there is no fallback if the explicit save fails or is interrupted. A short-lived cache keyed by `(entityType, uuid)` (in-memory, optionally mirrored to `sessionStorage`) would let us restore unsaved drafts on re-entry and let auto-save be best-effort rather than load-bearing. Becomes important once suggestion mode auto-saves on Next; useful more broadly as the app gains paths that move between editors. See `specifications/prompting.md` and `references/plans/suggestion-mode.md`.
- **Flush live-field saves on tab close/refresh** (`packages/frontend/src/hooks/useLiveFieldSave.ts`): `useLiveFieldSave` flushes pending debounced saves on component unmount (SPA navigation), but a hard browser close or page refresh fires before React can unmount. To cover this, add a `beforeunload` listener that calls `navigator.sendBeacon()` with the pending PATCH payload — `sendBeacon` is fire-and-forget and is explicitly permitted during page unload. Requires serialising the pending value and URL at `onChange` time. Low priority: the debounce window is short and users rarely close mid-edit, but worth closing if data loss reports surface.
- **Resilient action-log append failures** (`packages/api/src/commands/`, `packages/storage/src/action-log/`): v1 catches log-append failures and emits an `error`-level structured log only — entries that failed to write are lost from the canonical log file. The spec (`specifications/action-log.md` § Failure handling) describes the intended behavior: write the failed entry to a sidecar `<vault>/.maskor/action-log.failed.jsonl` so it is preserved and inspectable, and surface a low-key banner on the history page when failed entries exist in the session. Implement once v1 is shipped and the writer is stable. Adds two pieces: a sidecar writer (mirrors the main writer interface, simpler — no rotation needed) and a session-level "failed-append count" exposed via a small endpoint or via the existing health surface, plus a banner component on the history page.

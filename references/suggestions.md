# Suggestions

- **Add MSW for frontend API mocking** (`packages/frontend/src/test/`): The testing framework is set up but MSW (Mock Service Worker) is not yet installed. Once the frontend starts making API calls, add `msw` and create handler fixtures in `src/test/handlers.ts` so tests intercept HTTP at the network level rather than mocking `fetch` manually. Install: `bun add -D msw --cwd packages/frontend`.
- **`consumeAll` delete reliability** (`packages/storage/src/backend/markdown/vault.ts`): The piece file deletion in `consumeAll` calls `Bun.file().delete?.()` then falls back to `fs.unlink`. The optional chaining means a silent no-op if `delete` is absent. Consolidate to just `fs.unlink` for reliability.
- **Add CI workflow** (`.github/workflows/test.yml`): No CI is configured yet. A minimal GitHub Actions workflow running `bun install && bun run test && bun run typecheck` on push/PR would catch regressions automatically. See the sketch in `references/plans/testing-framework.md` section 7.
- **`contentHash` computation in `POST /fragments`** (`packages/api/src/routes/fragments.ts:136`): The handler sets `contentHash: ""` on every created fragment. Once the indexer or sequencer uses `contentHash` for change detection, empty hashes will cause false negatives. Compute a real hash (e.g. `Bun.hash(content).toString()`) before downstream consumers rely on it.
- **Update `ARCHITECTURE.md` schema table** (`references/ARCHITECTURE.md`): The Zod schemas table still lists the old names — `AspectSchema`, `NoteSchema`, `ReferenceSchema`. Each now has an `Indexed*` variant for list endpoints and a full variant for single-get. Update the table to reflect this split.
- **Write endpoints for aspects/notes/references** (`packages/api/src/routes/`): Storage supports write for all three, but they were deferred in milestone 1. Add `POST /` and `DELETE /:id` handlers when the frontend needs them — don't scaffold ahead of use.
- **Port/environment config** (`packages/api/src/index.ts`, `packages/frontend/`): API defaults to `3001` via `process.env.PORT`. When wiring the frontend, ensure both agree on port — consider a shared `.env` file at the repo root or a workspace-level config package to avoid silent mismatches.
- **Wire pino logger into the API** (`packages/api/src/app.ts`): `@maskor/shared` exports a pino logger factory. Add a single logger instance in `createApp` and pass it to route handlers for structured request/error logging before the frontend is wired up.
- **`vaultPath` discovery for frontend** (`packages/api/src/routes/projects.ts`): `POST /projects` expects a raw filesystem path in the body. The frontend (Tauri/Electron) must know the path to supply it. Deferred options: expose a file-picker over IPC, add a `/projects/discover` endpoint, or drive from a config file. Resolve before frontend integration begins.
- **Incremental watcher sync** (`packages/storage/src/index/`): `rebuild()` does a full O(n) file scan on every call. The next step is a file-watcher layer (chokidar) that calls targeted upserts on add/change and soft-deletes on unlink — replacing the full rebuild for live sessions.
- **`rebuild()` loads all vault entities into memory** (`packages/storage/src/index/indexer.ts`): The two-phase approach (read all → write in transaction) holds all fragment/aspect/note/reference entries in memory simultaneously. For very large vaults this may become a concern. Consider chunked or streaming writes as a follow-up if memory pressure is observed.
- **Consider partial indexes on `deleted_at IS NULL`** (`packages/storage/src/db/vault/schema.ts`): Composite and single-column indexes on `(pool, deleted_at)` and `deleted_at` are now in place. If the vault grows large and most rows are active, a partial index (`WHERE deleted_at IS NULL`) would be more space-efficient — SQLite supports this via raw SQL but not yet via the Drizzle schema DSL.
- **Implement manifest-based DB recovery** (`packages/storage/src/registry/registry.ts`): If the registry DB is lost or corrupted, the `.maskor/project.json` manifests inside each vault cannot currently be used to re-register projects. Add a `recoverFromManifests(vaultRoots: string[])` function that scans vault directories, reads manifests, and re-inserts missing rows. This makes the "self-describing vaults" promise from `SYNC_CONTRACT.md` real.
- **Promote API response types to `packages/shared` or a new `packages/api-types`** (`packages/frontend/src/api/types.ts`): The frontend integration plan inlines types that mirror the Zod schemas in `packages/api/src/schemas/`. These will drift. Once the frontend is wired, move them to a shared location — either `packages/shared` (if they are domain types) or a new `packages/api-types` (if they are presentation/transport types). Keeping them inline is a temporary compromise.
- **Commit generated API client** (`packages/frontend/src/api/generated/`): The orval codegen output is not yet in git. Commit it so CI doesn't require a running API server to build the frontend. Run `bun run codegen` after any API route change, then commit the diff.
- **Add a saved OpenAPI snapshot for offline codegen** (`packages/frontend/src/api/openapi.json`): Currently `bun run codegen` requires the API to be running at `localhost:3001`. Save a snapshot with `curl http://localhost:3001/doc > src/api/openapi.json` and point `orval.config.ts` at the file for offline / CI use. Switch back to the live URL during active API development.
- **Raw async functions from orval are PascalCase** (`packages/frontend/src/api/generated/`): The `operationName` capitalizer makes type names PascalCase but also affects the raw async functions (`ListFragments`, `GetProject`, etc.). These are unconventional for JS variables. If it becomes a problem, look into a custom `transformer` in orval config that renames only the type aliases post-generation while keeping function names camelCase.

# UUID revival on return from outside the vault

When a vault entity file disappears (`unlink`) and a file carrying the same UUID later reappears (`add`) anywhere in the same entity-type subtree, the watcher revives the original entity — clearing `deletedAt`, updating `filePath`, and upserting content — instead of treating the UUID as a collision and assigning a new one. Identity is anchored to the UUID, not the path; users can drag aspects/notes/references out of the vault, edit them externally for any length of time, and drag them back without losing the original entity's identity or the fragment-frontmatter references that point to its key.

A genuine collision (the DB-recorded file still exists on disk *and* a second file claims the same UUID) keeps the existing behavior: assign a new UUID to the newcomer, warn. Cross-entity-type returns (e.g. `aspects/x.md` → `notes/x.md`) intentionally do **not** revive — the original is soft-deleted, the destination becomes a fresh entity. Aspect→note flipping is reclassification, not preserved identity, and silently rewriting fragment-frontmatter weight references to note attachments would violate the "Maskor never rewrites fragment files" posture.

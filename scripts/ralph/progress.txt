## Codebase Patterns
- Vault skeleton dirs: fragments/, fragments/discarded/, aspects/, notes/, references/, pieces/
- `resolveProject` in storage-service is the entry point for first project access — good place for startup repairs
- Registry tests use `BASIC_VAULT` fixture (cpSync'd to tmp dir); storage-service tests do the same
- `ensureVaultSkeleton` uses `mkdir(path, { recursive: true })` making it idempotent and safe anywhere
- Fragment rename logic lives in `storage-service.ts` `fragments.write` — captures oldFilePath before write, then deletes old file if key changed
- Use `readdirSync` (not `existsSync`) to check exact on-disk filename casing — `existsSync` is case-insensitive on macOS and will return true for wrong-case paths
- Case-only rename fix: temp-file roundtrip (rename old→temp, write new, unlink temp); works on both case-sensitive (Linux) and case-insensitive (macOS/Windows) filesystems
- Rebuild-state tracking lives in `packages/api/src/middleware/rebuild-state.ts` (module-level Maps/Sets); import `isProjectRebuilding` for status checks, `registerRebuild` for deduplication
- Generated frontend API client is gitignored — always run `bun run codegen` in packages/frontend after adding/changing API endpoints
- To add a new path alias in frontend: update tsconfig.app.json paths, vite.config.ts alias, AND vitest.config.ts alias — all three are required
- Status endpoints that must NOT await rebuild: register in `projectsRouter` (under `/projects` prefix), not in `projectScopedApp` (which uses `resolveProject` middleware)
